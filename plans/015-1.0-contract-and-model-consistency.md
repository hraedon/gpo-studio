# Plan 015 — 1.0 contract and model consistency

Status: completed
Scope: define the supported 1.0 product and make every supported policy field
participate consistently in validation, hashing, diff, revision, import, and export
Depends on: Plan 014

> **Keystone — treat as a freeze point.** Plans 016 through 031 depend on the
> canonical schema, the split hashes (`policy_semantic_sha256` /
> `review_model_sha256`), and the golden vectors established here. Land this plan
> before opening downstream work, and once its golden vectors exist, revise them
> only under a deliberate, versioned migration — casual churn here cascades into
> every later plan.

## Purpose

GPO Studio has a strong registry-policy core, but recent capabilities were added
faster than the cross-cutting contracts around them. In particular, changing a
GPP collection currently does not change `semantic_sha256`, and the diff engine
does not report GPP or GPO-level changes. A digest or diff that omits exported
policy content cannot be used as a review or approval boundary.

The 1.0 product is an offline, single-operator authoring and review workbench.
It supports raw registry policy, ADMX-backed registry policy, links, security
filters, WMI filters, and the GPP Groups and Registry subsets with the six ILT
predicates implemented in Plan 014. It imports gpo-lens estate snapshots and
single-GPO GPMC backups and emits deterministic Studio and GPMC artifacts.

Live AD/SYSVOL writes, full GPMC parity, RSoP simulation, authentication, and
the additional GPP CSEs are explicitly post-1.0.

## WP-1 — Publish a capability contract

- Replace the stale roadmap table with `supported`, `preview`, `preserved`,
  `blocked`, and `out of scope` states.
- Document fidelity separately for authoring, import, export, PowerShell plan,
  diff, and Windows-lab verification.
- State that unknown CSE content is inventoried and hashed but cannot be edited
  or re-emitted until its bytes are preserved safely.
- Label the PowerShell plan accurately: registry, links, security filtering,
  and side status are actionable; WMI assignment and GPP are not applied by it.
- Add a support policy for Python versions, browsers, and workspace schema.

## WP-2 — Define complete canonical representations

- Introduce an explicit canonical schema version independent from the bundle
  schema version.
- Split hashes if necessary:
  - `policy_semantic_sha256` covers every field that changes effective policy
    or publication intent, including GPP and ILT;
  - `review_model_sha256` additionally covers review-relevant annotations and
    preserved-content metadata.
- Add canonical helpers for security filters, WMI filters, GPP collections,
  GPP elements, ILT predicates, side enablement, and domain/link intent.
- Define ordering and case-normalization rules for each collection. Preserve
  ordering where Windows semantics depend on it; sort only set-like data.
- Decide and document whether GUID/name are deployment identity or metadata.
- Add invariant tests proving that every export-affecting field changes the
  policy hash and that non-semantic revision timestamps do not.
- Add frozen golden vectors so other implementations can reproduce digests.

## WP-3 — Make validation exhaustive

- Add `validate_gpp_collection`, `validate_gpp_group`,
  `validate_gpp_registry`, and `validate_ilt_filter` dispatch functions with
  `assert_never()` on closed variants.
- Validate action/type combinations, required names/SIDs, registry hives and
  values, duplicate identities, XML-safe text, lengths, IP ranges, WQL shape,
  and GPP/side mismatch.
- Reject `cpassword` structurally and at every import/export boundary.
- Validate imported and restored snapshots before making them current.
- Make `ready` a guarded transition: a GPO cannot enter it with errors,
  unknown CSE content, unresolved conflicts, or unsupported preview content.
- Return stable issue codes and paths suitable for browser field mapping.

## WP-4 — Complete two-way and three-way diff

- Add typed changes/conflicts for metadata, side enablement, domain, GPP
  collections/elements, ILT, and preserved CSE metadata.
- Ensure security-filter equality includes SID as well as principal,
  permission, inheritance, and target type.
- Define stable identities for GPP elements. Do not use generated UI IDs as
  semantic identity.
- Detect link conflicts in three-way diff; current link output reports draft
  changes without comparing simultaneous observed changes.
- Render every new change and conflict in the browser and exported review
  report; never silently omit an unknown diff variant.
- Add convergence, delete/delete, edit/delete, reorder, and case-normalization
  tests for every supported kind.

## WP-5 — Fix numeric and mutation contracts

- Represent DWORD/QWORD request values as validated decimal strings at the
  JSON boundary, then convert to Python integers after range checking. This
  prevents browser `Number` precision loss for QWORD values above `2^53-1`.
- Apply the same contract to ADMX decimal and enum elements.
- Reject fractional, exponent, signed, padded, or otherwise non-canonical
  forms unless the contract explicitly permits them.
- Centralize mutation validation so API and store callers cannot diverge.
- Either use HTTP `If-Match` consistently or document the JSON
  `expected_revision` contract; do not claim both.
- Add tests for `0`, maxima, overflow, and the `2^53` boundary through the real
  browser/API serialization shape.

## Acceptance gates

- A GPP or ILT change changes `policy_semantic_sha256`.
- Every supported model field is covered by validation, two-way diff, and
  three-way conflict tests.
- Link and GPP concurrent edits produce conflicts rather than silent merges.
- Max QWORD round-trips from browser-shaped JSON without precision loss.
- `ready` cannot be set when release-blocking validation issues exist.
- Capability documentation matches executable endpoints and export behavior.
- Existing deterministic artifacts remain stable or change under a deliberate,
  versioned schema migration with golden fixtures.

## Sequence

WP-1 fixes the contract first. WP-2 and WP-3 establish canonical/validation
semantics. WP-4 consumes those semantics. WP-5 may proceed alongside WP-4 but
must land before any browser authoring expansion.
