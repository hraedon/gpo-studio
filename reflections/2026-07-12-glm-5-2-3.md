## On the project

GPO Studio now has comprehensive validation covering all model fields:
registry settings (type, range, key format, control characters), links
(DN format, order), security filters (principal format, duplicates),
and WMI filters (name, query syntax). The diff module correctly detects
all field changes including `target_type`. The import path now validates
the full GPO, not just settings.

The architecture continues to hold up well. The validation module is the
single source of truth for preflight checks, called by both the API layer
and the import path. The diff module is independent and testable. The
export module is clean with no dead code.

The codebase is now ~4800 lines across 16 source modules plus ~6400 lines
of tests (322 tests). The test-to-code ratio remains healthy.

## On the work done

**What went well:** The two-agent parallel dispatch was effective. Agent 1
handled all validation.py changes (WP-2, WP-3, WP-4) while Agent 2 handled
diff.py and export.py (WP-1, WP-5) with zero file conflicts. The adversarial
review was again the highest-value step — it found that the import path in
`api.py:import_backup` was constructing a `temp_gpo` with only `guid`, `name`,
and `settings`, completely omitting `security_filters` and `wmi_filter`. This
meant all the new validation checks would be bypassed for imported backups.
The fix was a one-line change to include those fields in `temp_gpo`.

**What was awkward:** The duplicate principal detection initially used
`principal.casefold()` instead of `stripped.casefold()`. This meant
`"Domain Admins"` and `" Domain Admins "` (with whitespace) were not
detected as duplicates. The fix was straightforward but the mistake is
a reminder that normalization should happen before comparison, not after.

The WMI query validation initially used substring matching
(`"select" not in query.casefold()`), which would accept
`"selection fromage"` as valid WQL. The fix uses word-boundary regex
(`\bselect\b`, `\bfrom\b`). This is still just a syntax check, not a
full WQL parser — but it's sufficient to catch obviously malformed queries.

**What I'm confident in:** The `_security_filters_equal` fix is correct
and well-tested. The validation checks are comprehensive and cover all
the gaps identified in the Plan 007 reflection. The import path now
validates the full GPO. The dead code removal in export.py is clean —
`type_map[setting.registry_type]` is safe because `RegistryType` is a
`Literal` that exactly matches the dict keys.

**What I'd want a second pair of eyes on:** The WMI query validation is
a keyword check, not a WQL parser. A query like
`SELECT garbage FROM` would pass. But since the query is never executed
(it's stored and emitted as a comment in the PowerShell plan with
newlines sanitized), the risk is very low. A full WQL parser would be
over-engineering for v0.1.

## On what remains

**Needed before this can ship as a release:**

1. Split `studio.js` into multiple files or add module structure — the
   UI code is getting increasingly complex and the dense single-line
   style is becoming a maintenance burden.
2. The GPMC backup XML uses our own attribute-based format for
   SecurityFilter elements instead of the real GPMC schema structure
   (Trustee/Sid/Name/Type child elements). Requires a real SDDL/SID
   resolver.
3. PowerShell plan only replaces security filters per-target with
   `-Replace`. It doesn't remove targets that are no longer in the draft.
   A truly idempotent plan would need to query existing permissions
   and remove stale ones.
4. No security filter principal format validation (SID format,
   domain\user format). Current validation checks for empty, control
   chars, length, and duplicates — but not format.
5. No full WQL syntax validation (current check is keyword-based).

**Nice to have:**

6. SDDL string parsing for security filter editing.
7. Item-level targeting (GPP) support.
8. Loopback mode configuration.
9. ADMX `list` element editing in the UI.
10. File upload for backup import (v0.1 is local-only).
11. Multi-GPO backup import.
12. ADMX fixture with elements that omit `key` (testing key inheritance
    from parent policy).

## Gaps to flag

- `validation.py:validate_gpo` — The WMI query check uses word-boundary
  regex but is not a full WQL parser. Malformed-but-keyword-containing
  queries pass. Risk is low since queries are never executed.
- `api.py:import_backup` — The `temp_gpo` now includes security_filters
  and wmi_filter for validation, but the `domain` field is still taken
  from `backup_gpo.domain` without validation. A backup with a very long
  or control-character domain would pass validation. The API Pydantic
  layer catches this for direct mutations, but the import path doesn't
  go through Pydantic for the domain field.
- `export.py:_ps_sanitize_comment` — Only replaces `\n` and `\r`. Other
  characters that could theoretically affect PowerShell parsing (backtick,
  curly braces) are not sanitized. Risk is very low since values stay
  in comment lines. (Carried forward from Plan 007 reflection.)
- `backup.py:parse_manifest` — Uses `.//` XPath which searches all
  descendants, not just direct children. In malicious XML with nested
  GPO elements, an outer GPO could inherit an inner GPO's security
  filters or WMI filter. (Carried forward from Plan 007 reflection.)
- `tests/fixtures/admx/` — Fixtures still don't exercise policy-level
  `enabled`/`value` attributes or key inheritance from parent policy
  to child elements. (Carried forward from Plan 007 reflection.)
