## On the project

GPO Studio now has a complete, consistent cross-cutting contract layer. Every
supported policy field — raw registry, ADMX registry, links, security filters
(with SID), WMI filters, GPP Groups, GPP Registry, ILT predicates, side
enablement, domain, and preserved CSE metadata — now participates in the
canonical hash, exhaustive validation, two-way diff, and three-way conflict
detection. The split hashes (`policy_semantic_sha256` / `review_model_sha256`)
give a review/approval boundary that cannot omit exported content, and frozen
golden vectors lock the digest contract for downstream plans.

The numeric contract closes the QWORD precision hole: DWORD/QWORD values cross
the JSON boundary as canonical decimal strings in BOTH directions (request via
the SettingData validator + forms.mjs, response via numeric stringification),
so a max QWORD round-trips through a browser without precision loss. The
`ready` status transition is now a guarded gate enforced at every entry point
(update_metadata, import_baseline_gpos, restore_revision): a GPO cannot enter
ready with error-severity validation issues or unknown/preserved CSE content.

The codebase grew from ~5638 to ~6400 source lines (added numeric.py; canonical,
validation, diff roughly doubled) and ~7866 to ~9900 test lines.

## On the work done

**What went well:** Phased dispatch with disjoint file ownership kept conflicts
manageable. The two reviewers were the highest-value step and caught genuine
defects: the three-way GPP conflict sort crashed with TypeError on mixed
group+registry conflicts in the same scope; import/restore bypassed the ready
gate; the frontend `Number()` coercion defeated the entire QWORD contract; bool
slipped through DWORD validation (`isinstance(True, int)`); REG_SZ/BINARY type
gaps let export crash on data that passed validation.

**What was awkward:** Two of the three parallel agents in the second phase
(WP-4 diff, WP-5 numeric) both modified api.py, numeric.py, and policy_config.py
concurrently. The final filesystem state was coherent by luck (last-write-wins
produced a consistent merge), but this was fragile. The strict disjoint-ownership
discipline in the fix round was the corrective. Also, the WP-1 docs agent
substantially overstepped its scope and also implemented WP-3 validation — the
work was high quality and 545 tests passed, but it required careful verification
that the unsolicited code was correct before building on it.

**What I'm confident in:** The canonical hash now changes on every export-affecting
field mutation (proven by invariant tests) and is stable across insertion order
and case-only changes. The golden vectors are recomputed and frozen. The diff
engine reports every supported kind and never silently omits a variant. The
ready gate is enforced at all three mutation entry points. The numeric contract
round-trips max QWORD exactly.

**What I'd want a second pair of eyes on:** The response-side QWORD
stringification transforms `gpo.to_dict()` output in `_gpo_payload` and
`list_gpos` only — other response paths (e.g. revision snapshots, three-way diff
payloads) still emit ints. If a browser reads QWORD from those paths it could
lose precision; this is acceptable for 1.0 (those paths are not the primary
authoring round-trip) but worth noting. The three-way CSE-metadata conflict
detection mirrors the pattern but CSE metadata is `preserved` (read-only), so
real conflicts are unlikely in practice.

## On what remains

**Needed before 1.0 (Plans 016–020):**

1. **Plan 016 — GPP end-to-end authoring.** GPP/ILT survive storage and backup
   round trips and now diff/hash/validate correctly, but there are no GPP
   mutation endpoints or browser editors. A feature is not 1.0 until a user can
   import/inspect/edit/validate/diff/restore/export it without editing JSON or
   SQLite.
2. **Plan 018 — Workspace and runtime hardening.** The store has WAL + SQL CAS
   but no schema-version framework, recovery commands, busy policy, concurrent
   stress suite, or bounded-input budgets. These are 1.0 durability/security gates.
3. **Plan 017 — Windows interoperability program.** Requires a disposable AD
   lab; Python round-trips are not proof GPMC/Windows accepts the artifacts.
   Environmental dependency — cannot be done in this environment.
4. **Plan 019 — Browser quality and accessibility.** No JS/browser/accessibility
   test suite yet; concurrency/failure UX and review-centered workflows need work.
5. **Plan 020 — Release engineering and 1.0 gates.** No tags, changelog,
   coverage/fuzz/browser gates, SBOM, or installed-wheel smoke test.

**Deferred to the relevant plan:**
- GPP `semantic_dict` uses the general GPP namespace, not type-specific
  (Groups/Registry) namespaces — verify against real GPMC in Plan 017.
- SDDL editor UI, signed publication bundles, loopback mode — post-1.0.

## Gaps to flag

- `api.py:_stringify_numeric_settings` — Only applied to `_gpo_payload` and
  `list_gpos`. Three-way diff payloads and revision-snapshot responses still
  emit DWORD/QWORD as JSON numbers. Low risk for 1.0 (not the primary authoring
  path) but a precision edge case if a browser reads QWORD from those paths.
- `diff.py:_diff_gpp` — ILT equality is order-sensitive and negate-sensitive
  (correct), but there is no standalone ILT diff test exercising reorder/negate
  in isolation; ILT is only tested as part of GPP element equality.
- `canonical.py:gpp_registry_value_identity` — Includes `registry_type` in the
  identity tuple, so a type change (e.g. REG_SZ → REG_DWORD on the same value
  name) appears as remove+add rather than modify. This diverges from Windows
  registry semantics where a value is identified by name alone. Debatable;
  flagged for Plan 016/017 review.
- `store.py:import_baseline_gpos` — Now rejects invalid GPOs (counts under
  `rejected`); estate/test consumers that assert exact summary dict equality
  were updated, but any external caller expecting the old 4-key dict shape will
  break (now 5 keys).
- The `semantic_sha256` manifest key was removed (deliberate versioned change);
  `schema_version` stayed at 2. If a 1.0 consumer expects schema_version to
  bump on key changes, bump it in Plan 020.
