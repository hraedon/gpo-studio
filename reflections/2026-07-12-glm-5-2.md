## On the project

GPO Studio is getting close to being a usable authoring tool. Plan 006
closed the most significant export and diff gaps: security filters and
WMI filters are now in the PowerShell plan (with correct cmdlet
parameters after the adversarial fix), in the GPMC backup XML, and in
the diff engine with three-way conflict detection. The symlink hardening
closes the most dangerous inbox attack vector.

The architecture continues to hold up well. The layering
(model → store → domain → API → UI) is clean, frozen dataclasses with
strict mypy catch real bugs, and the safety boundary (no AD writes) is
intact. The codebase is now ~4000 lines across 16 source modules plus
~5500 lines of tests.

The strongest aspect remains the adversarial review process. Both
reviewers found the same three CRITICAL bugs independently: the
PowerShell `Set-GPPermissions` (plural, deprecated) was missing the
mandatory `-TargetType` parameter and had a nonexistent `-Inheritable`
parameter; the `Set-GPInheritance` cmdlet was being used for WMI filter
assignment when it actually manages inheritance blocking on containers;
and a test was accidentally merged during the WP-6 subagent's work.
These are the kind of bugs that would have made the generated apply.ps1
completely non-functional in a real AD environment.

The weakest aspect is still the UI code (not touched in this plan) and
the GPMC backup XML format. The XML SecurityFilter elements use
attributes (principal, permission, inheritable) instead of the real GPMC
schema structure (Trustee/Sid/Name/Type child elements). This is
documented as deferred — making it GPMC-compatible would require a
real SDDL/SID resolver. The current format is our own metadata that
won't round-trip through a real GPMC, but is useful for studio-to-studio
transfer.

## On the work done

**What went well:** The four-subagent parallel dispatch was very
effective. All six work packages were independent, so all four subagents
ran in parallel with no file conflicts. The adversarial review was again
the highest-value step — it caught bugs that would have shipped and made
the PowerShell plan non-functional.

**What was awkward:** The WP-6 subagent accidentally merged the
`test_security_filter_add_edit_delete` test function body into the new
`test_import_backup_rejects_symlinked_manifest` test, creating one
monster test that tested two unrelated things. Both adversarial
reviewers caught this. The fix was straightforward but the subagent
should have been more careful about which test function it was replacing.

The PowerShell cmdlet correctness issue was the biggest learning. The
initial implementation used `Set-GPPermissions` (plural, deprecated)
with `-Inheritable` (nonexistent parameter) and `Set-GPInheritance`
with `-WmiFilter` (wrong cmdlet, nonexistent parameter). The plan
documented the wrong cmdlet for WMI filter assignment. The fix: use
`Set-GPPermission` (singular) with `-TargetType Group -Replace`, and
emit a comment for WMI filter assignment since there's no standard
PowerShell cmdlet for it.

**What I'm confident in:** The diff engine extension is correct —
security filter changes match by principal.casefold(), WMI filter changes
compare all four fields (name, query, language, description), and
three-way conflict detection follows the exact same pattern as the
existing settings conflict detection. The symlink hardening is robust:
`os.scandir(follow_symlinks=False)` replaces `rglob("*")`, manifest.xml
and bkupInfo.xml are checked for symlinks before reading, and the GPO
directory itself is checked. The canonical hash now includes
wmi_filter.description, fixing the inconsistency where the diff
detected description changes but the hash didn't.

**What I'd want a second pair of eyes on:** The `_safe_path` function in
`backup.py` still has a symlink detection gap — `resolve()` follows
symlinks, so `resolved.is_symlink()` returns False for symlinks pointing
to existing targets. This is flagged as Plan 007 WP-3. The TOCTOU race
between `is_symlink()` checks and file reads is also flagged as Plan 007
WP-4 (using `os.O_NOFOLLOW`).

## Gaps to flag

- `backup.py:_safe_path` — `resolve()` follows symlinks, so
  `resolved.is_symlink()` returns False for symlinks pointing to existing
  files. A symlink within the backup directory pointing to another file
  within the backup directory passes both checks undetected. Plan 007
  WP-3 addresses this.
- `backup.py:_hash_file` and `read_backup` — TOCTOU race between
  `is_symlink()` check and `open()`/`read_bytes()`. An attacker with
  local filesystem access could replace a regular file with a symlink
  between the check and the read. Plan 007 WP-4 addresses this with
  `os.O_NOFOLLOW`.
- `export.py:powershell_plan` — The `-TargetType Group` is hardcoded
  since the SecurityFilter model doesn't store the trustee type. An
  admin must manually adjust for User/Computer targets. Plan 007 WP-5
  adds the field.
- `export.py:gpmc_backup_bundle` — SecurityFilters XML uses attributes
  instead of GPMC schema elements (Trustee/Sid/Name/Type). Not
  GPMC-compatible. Would need a real SDDL/SID resolver. Documented as
  deferred.
- `api.py:import_backup` — Does not pass security_filters or wmi_filter
  from the backup to `create_gpo`. Security filters and WMI filter from
  backups are silently lost on import. Plan 007 WP-1 and WP-2.
- `canonical.py:semantic_dict` — Now includes `wmi_filter.description`
  (fixed this session), but the `wmi_filter.id` is still excluded. This
  is intentional (id is implementation detail) but should be documented.
- `diff.py` — `SecurityFilterChange` and `WmiFilterChange` are new
  dataclasses that extend `TwoWayDiff` and `ThreeWayDiff`. Existing API
  consumers that call `asdict(result)` will now see additional fields.
  This is backward-compatible (new fields have defaults) but changes the
  response shape.
- `store.py:create_gpo` — Now accepts `domain` parameter (added this
  session), but still doesn't accept `security_filters` or `wmi_filter`.
  Plan 007 WP-6.
- `tests/fixtures/admx/` — Fixtures don't exercise policy-level
  `enabled`/`value` attributes or key inheritance from parent policy to
  child elements. Real Microsoft ADMX files sometimes omit `key` on
  elements, expecting inheritance from the parent `<policy key="...">`.
  The parser returns `""` for missing key, which would produce incorrect
  `registry_key`. This gap is untested.
