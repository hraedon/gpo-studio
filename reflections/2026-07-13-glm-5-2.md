## On the project

GPO Studio now supports **end-to-end GPP authoring through the browser**. A user
can create a draft GPO, switch to the Preferences tab, add/edit/delete GPP
Groups (with ordered members), Registry items (with typed values), and ILT
targeting predicates — all through dialogs, no JSON/XML editing. Each action
creates exactly one immutable revision; the policy hash, validation, and diff
all reflect GPP/ILT content. The ILT editor supports all six predicate types
(OU, group, registry, IP range, environment, WMI query) with negation and an
AND-only combinator (labeled limitation), a live plain-language preview, and a
guard against silently dropping unsupported imported predicates.

The frontend is dependency-light vanilla ES modules (gpp.mjs ~220 lines), matching
the established dense style. The backend WP-1 API (reviewed across four Sol
rounds) provides full typed CRUD for groups/registry/members/values with
deterministic editor IDs, exhaustive validation, and order-aware diff/conflicts.

## On the work done

**What went well:** The frontend agent ran a full 12-step Playwright smoke test
(zero console errors) before reporting — exactly the right verification given no
JS suite exists. The adversarial reviewer caught the one genuinely dangerous
defect: importing a GPP item with an unsupported ILT predicate type silently
rewrote it to "ou" on edit (data corruption). The fix marks unsupported
predicates read-only and confirms before dropping them.

**What I'm confident in:** The numeric contract is intact — DWORD/QWORD cross
the boundary as decimal strings in both directions, MULTI_SZ splits on
newlines+semicolons, and blank numerics are caught client-side before the 422.
All innerHTML interpolation is escaped. The CRUD wiring (POST creates fresh,
PUT is must_exist, DELETE confirms) matches the established pattern. Post-
mutation refresh (state/validation/hash/renderAll) is consistent.

**What I'd want a second pair of eyes on:** The ILT unsupported-predicate guard
uses `confirm()` before dropping — functional but not the plan's ideal ("render
read-only, block export after lossy edit"). A fuller treatment needs the backend
to expose unknown-content flags (WP-2). The ILT editor is AND-only; complex
imported boolean expressions are not yet rendered read-only as a whole (only
per-predicate-type).

## On what remains (Plan 016)

**Needed to finish Plan 016:**

1. **WP-2 — Fidelity / unknown-content preservation.** The core gap: the GPP
   parser drops unknown XML attributes/elements on round-trip (lossy). The plan
   says "preserve unknown XML attributes and ILT predicates losslessly when an
   item is not edited; if lossless preservation is not implemented, mark the item
   read-only and block export after a potentially lossy edit." This needs a
   preserved-attributes bag on the GPP dataclasses + parser/serializer round-trip
   + a read-only/lossy indicator surfaced to the UI. Architecturally significant;
   could churn the canonical contract — do deliberately.
2. **WP-5 — End-to-end browser tests.** No browser automation suite exists yet
   (Plan 019 scope). The frontend is currently verified by manual Playwright
   smoke tests, not committed CI. A committed Playwright suite covering the GPP
   authoring journeys (create/edit/delete, all registry types, all ILT
   predicates, stale-revision recovery) is the 1.0 gate.

**Deferred:** Clone/reorder/restore-from-revision row actions (WP-3 mentions
these); they are enhancements over the core edit/delete flow.

## Gaps to flag

- `gpp.mjs` ILT editor — AND-only; imported OR/grouped expressions are not
  preserved or rendered read-only as a unit. Per-predicate-type unsupported
  guard exists, but boolean-structure preservation is absent.
- `gpp.py` parser — Unknown XML attributes (uid, desc, bypassErrors,
  userContext, etc.) and unknown ILT filter element types are silently dropped.
  This is the WP-2 lossless-preservation gap. Until fixed, editing an imported
  GPP item is potentially lossy for any field the model doesn't capture.
- No committed browser test suite — frontend correctness relies on ad-hoc
  Playwright smoke tests run by agents, not CI. Plan 019 (browser quality) or
  WP-5 should establish this.
- `api.py` response models — `gpo` is now fully typed via `GpoResponse`, but the
  `value` fields are `str | list[str]` (post-stringification). Correct for the
  response shape, but a consumer must know DWORD/QWORD arrive as decimal strings.
