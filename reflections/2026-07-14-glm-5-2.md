**Work summary:** Resolved all 4 items from sol's feedback review: P0 (registry-level metadata broadening to every value), P1 (coalescing losing item identity/order, key-only entries rejected by API/browser), P2 (root metadata changes missing from three-way conflict detection). Two rounds of adversarial review (kimi + glm) addressed critical/major issues before commit.

## On the project

The GPP registry model is getting more complex as it correctly handles real-world MS-GPPREF XML — key-only entries, per-value ILT, non-consecutive same-key elements. The data model (GppRegistry with multiple GppRegistryValue) is an abstraction over the XML's one-Properties-per-Registry structure, and that abstraction is leaking. The diff system's use of `gpp_registry_identity` (hive\key) as a dict key is a latent collision bug now that non-consecutive same-key entries are preserved as separate GppRegistry objects. This didn't cause test failures but will bite when real-world data with interleaved keys is diffed.

The separation between API validation (Pydantic) and domain validation (validation.py) creates a risk of inconsistency — we had to carefully align key-only detection logic (`name == ""` vs `not name.strip()`) across both layers. The AGENTS.md directive to use `assert_never()` in dispatch is well-followed and caught nothing this session but provides good safety.

## On the work done

**Confident in:**
- P0 ILT broadening fix: only applying registry-level metadata to the first value across all three layers (dict loading, API, XML parser) is correct and well-tested.
- P1 key-only entries: allowing empty name/type/value through the full stack (API model, validation, browser, parser, serializer) with proper round-trip tests.
- P2 three-way conflict detection: the `_three_way_gpp_collection_conflicts` function follows the exact pattern of other conflict detectors and is tested with convergent, draft-only, and divergent cases.

**Less confident in:**
- Browser ILT handling: the common ILT editor only shows when all values have identical ILT. When the user adds a common ILT to a registry with mixed per-value ILTs, it overwrites all of them. This is a user action (not automatic broadening), but data loss is silent. The reviewers flagged this; I judged it acceptable since the editor is empty when values have different ILTs.
- The `_parse_registry` change to only apply element-level metadata to the first value in `props_list` (legacy multi-Properties format). This is correct for MS-GPPREF (one Properties per Registry), but the legacy Studio format with multiple Properties is non-standard and the behavior is "least surprise" rather than tested.

## On what remains

- **Diff system identity collision** (pre-existing, now more likely to trigger): `_diff_gpp_registry` uses `gpp_registry_identity` (hive\key) as a dict key. Multiple GppRegistry objects with the same key will collapse in the diff map. Needs a position-aware or value-aware identity scheme. Not blocking but should be tracked.
- **Browser UX for per-value ILT**: there's no visual indication that per-value ILTs exist when the common editor is empty. A warning or per-value ILT badge would help.
- **Canonical hash inconsistency**: key-only entries parsed from XML have `registry_type=""`, but API-created key-only entries default to `registry_type="REG_SZ"`. Both are valid but produce different canonical hashes. Should normalize.

## Gaps to flag

- `src/gpo_studio/diff.py:545-546` — `_diff_gpp_registry` uses `dict[str, GppRegistry]` keyed by `gpp_registry_identity`, silently collapsing non-consecutive same-key entries. Exposed by the P1 coalescing fix.
- `src/gpo_studio/api.py:219` — `registry_type` changed from `Literal` to `str` for key-only support. Runtime validation compensates but OpenAPI schema lost type safety. Could use `Literal["", "REG_SZ", ...]` instead.
- `src/gpo_studio/static/js/gpp.mjs:199` — `values.forEach(v=>{v.ilt_filter=commonIlt})` overwrites per-value ILTs when a common ILT is set. No confirmation dialog for this data loss path.
- `src/gpo_studio/canonical.py:134` — `gpp_registry_value_identity` returns `(name.casefold(), registry_type)`. Key-only entries with `registry_type=""` vs `"REG_SZ"` produce different identities/hashes for semantically identical entries.
- `src/gpo_studio/api.py:1935-1937` — Import path creates `temp_gpo` without `gpp_collections`, so `validate_gpo` doesn't validate GPP collections during import. The grouped-invalid representation issue from the feedback is fixed by the coalescing change, but GPP collections are still not validated on import.
