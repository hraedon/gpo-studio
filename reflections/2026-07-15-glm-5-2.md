## On the project

GPO Studio's security architecture is now substantially hardened. The staging-file pattern Sol introduced is the right call — it eliminates an entire class of TOCTOU by copying to a private unpredictable path before any SQLite operation, then atomically publishing. The `safe_io` module is genuinely impressive: the two-phase NtOpenFile + CreateFileW walk with FILE_ID_INFO verification is the correct way to resist path substitution on Windows, and the `iter_directory` NtQueryDirectoryFile implementation avoids `os.listdir` materialization entirely.

The main tension is Windows file locking semantics. POSIX allows renaming files with open handles; Windows doesn't. This means some race-condition tests that work on POSIX (parent swap during enumeration) can't be replicated on Windows because the directory handle pins the directory — which is actually the desired security property, just expressed differently.

## On the work done

**What went well:**
- The REG_MULTI_SZ fix was clean and self-contained. Checking `decoded.count("\0")` before splitting avoids materializing the list, and the cap now applies to decode, encode, and both validation paths.
- The backup enumeration fix (count all entries including directories, check batch size after `os.listdir`) is simple and correct.
- The Windows validation was essential. The `os.link` sharing violation was not discoverable on Linux — it only manifests because `CreateHardLinkW` refuses to create a link when the source file has any open handle, even with `FILE_SHARE_READ|FILE_SHARE_WRITE`. The fix (close staging fds before publish, don't hold directory fd on Windows) is correct.

**What I'm confident in:**
- All five P1 fixes address the stated vulnerabilities correctly.
- The Windows test results (1103 passed, 2 skipped) validate the NtCreateFile paths, lock contention, junction rejection, and SQLite backup/restore staging.
- The `FILE_ID_INFO` identity check is more reliable than `st_ino` on Windows and fails closed when unavailable.

**What I'd want a second pair of eyes on:**
- The `_publish_no_replace` function now has separate Windows and POSIX code paths. The Windows path doesn't fsync the parent directory (no `dir_fd` available). This is acceptable for a single-operator tool but worth noting.
- The `test_restore_uses_pinned_backup_after_public_path_substitution` test catches `PermissionError` on Windows, which means the substitution attempt is silently skipped. The test still verifies the correct data is restored, but the race window isn't exercised on Windows.

## Gaps to flag

- `src/gpo_studio/workspace_ops.py:202-230` — `_publish_no_replace` skips parent directory fsync on Windows. Acceptable for single-operator but not crash-safe against power loss.
- `src/gpo_studio/backup.py:558-587` — Windows backup enumeration still uses path-based `os.scandir` rather than the fd-based `iter_directory` from `safe_io.py`. The fd-based approach would be more race-resistant.
- `tests/test_workspace_ops.py:1317` — The `substitute_after_copy` test hook catches `PermissionError` on Windows, which means the TOCTOU race isn't actually exercised on Windows. The test still validates correct data, but the attack path is different (file pinning vs. staging).
- `tests/test_backup.py:771-802` — `test_scan_hashes_pinned_file_after_parent_swap` is skipped on Windows. The directory handle pinning prevents the rename, which is the correct security behavior, but the test doesn't verify this directly.
- `src/gpo_studio/api.py:2183` — Import error sanitization replaces `GppError`/`RegistryPolError` messages with a generic string. This is correct for security but loses diagnostic information. The Plan 018 doc notes "richer context could improve operator diagnostics without leaking paths."
