## On the project

GPO Studio's discipline is the thing that stands out. The "correctness by
construction" charter — `mypy --strict` plus `assert_never()` in every closed-set
default branch — is not decorative; it changes how you write the code. I hit it
directly: my first instinct for `_derive_classification` was a `match` over
`tuple[bool, bool]` with `assert_never` in the default, and mypy correctly
rejected it because `(False, False)` is type-reachable. That forced the honest
question "what SHOULD happen with no evidence?" → `unknown`, which is the right
answer. The charter made the code better, not just stricter.

The pre-review spike boundary (Plan 021 banner) is the most important
load-bearing idea in this whole program: it says the review gate exists *to
ratify* the inventory/evidence contract, so nothing contract-shaped (bundle
schemas, hash profiles, canonical vectors) may land on `main` first. This
session respected it — only documents and a generator shipped. The WP-1
inventory and WP-4 schema are deliberately pre-gate and versioned `0.1.0` /
provisional; almost every row is `unknown` because no endpoint evidence exists
yet. That's correct, not a gap to fill hurriedly.

## On the work done

**What went well:**
- The generator's core invariant — "no `verified-rw` without Windows AND endpoint
  passing evidence" — is enforced mechanically and proven by tests that try
  every combination of evidence kinds. A reviewer doesn't have to trust the
  docstring; the tests are the contract.
- Delegating the two large research-heavy docs to general subagents in parallel
  worked well. The WP-1 inventory's CSE GUIDs survived cross-lineage
  verification (all 11 spot-checked GUIDs confirmed against Microsoft sources)
  except one *label* (the AppLocker row pointed at the Code-Integrity-policy
  CSE GUID — the GUID was real, the name was wrong). That's exactly what
  adversarial review is for.

**What I'm less sure about:**
- The adversarial review's headline finding was real and serious: the WP-4
  doc's own example evidence pack **did not parse** against the loader I wrote
  (`side: "machine"` vs `computer`; integer `build`; missing `evidence_kind`;
  null `evidence_hash`). I fixed the doc to match the code, but the lesson is
  that the doc and code were authored by different subagents with no shared
  contract. The right structural fix is a test that extracts the doc's JSON
  block and parses it — I verified it by hand this session but did not commit
  that as an automated check. That's a real gap (see below).
- I chose to *soften* two doc claims rather than implement them: the
  `schema_version: 0` legacy adapter (now documented as out-of-scope) and the
  pinned-manifest refusal (now a documented WP-4 follow-up rather than a CLI
  feature). Both are honest, but they mean the doc now describes a generator
  that's intentionally a subset of the full WP-4 contract. The review gate
  should ratify whether that subset is acceptable or whether the pinned-manifest
  enforcement must ship before the gate.

**What I'd want a second pair of eyes on:**
- `canonical_pack_bytes` relies on `dataclasses.asdict` over `slots=True`
  dataclasses. It's deterministic and tested for stability, but the "two
  independent implementations reproduce canonical/hash vectors" acceptance gate
  is not yet demonstrated by a second implementation — only by my own round-trip.
  That gate item is still open by construction.

## Gaps to flag

- **No automated doc/code contract test.** `docs/plan-021/reference-estates-and-evidence.md`
  contains an example evidence pack that I verified parses against
  `src/gpo_studio/evidence.py:parse_pack` by hand, but there is no committed
  test asserting this. The adversarial reviewer caught a parse mismatch; a test
  would have caught it at authoring time. (`tests/test_evidence.py` — add a
  test that reads the markdown, extracts the ```json block, and asserts
  `parse_pack` succeeds.)
- **Pinned-manifest refusal is documented but not implemented.**
  `docs/plan-021/reference-estates-and-evidence.md` (Signing model) says a pack
  whose hash isn't in the pinned manifest "MUST NOT be treated as release
  evidence," but `scripts/generate_public_matrix.py` has no `--pinned-manifest`
  argument. The signable gate (redaction+licensing) IS enforced; the
  hash-pinning is not. This is documented as a follow-up but a reader of the doc
  alone would assume it's enforced.
- **`schema_version: 0` legacy claim softened, not removed.** The doc still
  references the 1.0 `release-evidence-report.json` as a "conceptual ancestor"
  of `schema_version: 0`, but `load_pack` rejects `!= 1`. There is no adapter.
  If someone tries to feed the 1.0 report to the generator, they get a hard
  error. That's acceptable for pre-gate but should be explicit in the doc
  (it now is, but only after the review fix).
- **`evidence.py` `_derive_classification` does not use `assert_never`.** I
  deliberately used an if/else instead of `match`+`assert_never` because the
  closed set is `bool × bool` (4 reachable combos), not an enum that could gain
  a variant. I judged this acceptable, but a strict reading of AGENTS.md ("every
  dispatch over a closed set") could want the match form. The four cases are
  all explicitly handled, so no variant can be silently missed. (`src/gpo_studio/evidence.py`,
  `_derive_classification`.)
- **Lab substrate unreachable this session.** `acb` was not on `mvmhermes01`
  (`/home/itadmin/.local/bin` had `agent-notes` but not `acb`). The brief says
  the windows-evidence-lab command boundary was live-qualified 2026-07-18 from
  the Linux operator box, but I could not exercise it. No fresh Windows
  evidence was gathered; the only verified rows in the inventory are carried
  forward from the 1.0 release-evidence (WS2025 registry, GPO create, side
  enablement — all `verified-ro`, not `verified-rw`).
- **WP-2/WP-3 architecture spikes deliberately not done.** The brief marks
  them optional and inside the spike boundary (throwaway branches only). I
  scoped them out to respect the boundary and because they're not needed for
  the gate. If the gate wants a lossless three-layer model sketch, that's a
  separate throwaway-branch session.
